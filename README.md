# Real Estate Observatory

Static observatory for French housing-market and income data.

The repository combines:

- a Vite / React / TypeScript frontend
- Python data pipelines for DVF and FiLoSoFi
- Cloudflare R2 for publishing Parquet artifacts
- DuckDB-Wasm in the browser for interactive querying

There is no application backend. The site is static. It reads:

- versioned JSON files from `public/data/`
- remote FiLoSoFi Parquet and metadata files published on R2

## What the project does

The project currently exposes three main product layers:

1. Public homepage and statistics pages with curated indicators and charts.
2. A Data Explorer page with:
   - a local tabular preview for DVF
   - browser-side FiLoSoFi queries executed through DuckDB-Wasm
3. Reusable data assets generated by the pipelines:
   - yearly DVF outputs
   - yearly and multi-year FiLoSoFi outputs
   - merged commune-year analytical outputs

## Repository structure

### Frontend

- `src/`: React application
- `public/data/`: JSON and GeoJSON files served directly by the static site
- `src/services/dataAssetConfig.ts`: builds remote public asset URLs from `VITE_DATA_ASSET_BASE_URL`
- `src/services/dataExplorerService.ts`: loads public previews and DVF year-specific previews
- `src/services/filosofiDataService.ts`: queries remote FiLoSoFi Parquet files with DuckDB-Wasm

### Data pipelines

- `data/scripts/dvf/`: DVF download and transformations
- `data/scripts/filosofi/`: FiLoSoFi download and transformations
- `data/scripts/publishing/`: preview and publication helpers
- `scripts/orchestration/`: pipeline entrypoints
- `scripts/storage/`: Cloudflare R2 upload helpers
- `scripts/shared/`: shared configuration helpers
- `config/`: years, source catalogues and canonical mappings

## Data architecture

The local pipeline is partitioned by dataset and year:

```text
data/
├── raw/<dataset>/year=YYYY/
├── bronze/<dataset>/year=YYYY/
├── silver/<dataset>/year=YYYY/
└── gold/<dataset>/year=YYYY/
```

FiLoSoFi also produces multi-year gold artifacts:

```text
data/gold/filosofi/
├── commune_all_years.parquet
├── department_official/department_all_years.parquet
├── department_derived/department_all_years.parquet
├── metadata.json
├── indicator_availability.json
└── summaries_by_year.json
```

The merged cross-source dataset is:

```text
data/gold/commune_year/commune_year.parquet
```

## Main sources

### DVF

- Producer: DGFiP / Etalab
- Scope used here: residential sales only
- Use in the project:
  - territorial price indicators
  - national and departmental price summaries
  - Data Explorer raw-like previews

Important limitation:

- DVF does not include Bas-Rhin, Haut-Rhin or Moselle, because these departments follow the local land-registry regime (`livre foncier`) rather than the standard DVF publication flow.

### FiLoSoFi

- Producer: Insee
- Use in the project:
  - income indicators
  - poverty indicators
  - commune and department exploration
  - national chart inputs for the homepage

Important limitations:

- `2022` is intentionally absent because Insee did not publish a FiLoSoFi vintage for that year.
- `2023` belongs to FiLoSoFi 2 and introduces a methodological break.
- Some variables are only available for some years and some geographies.

## Core pipeline choices and assumptions

These points are important because they explain what the numbers mean.

### DVF filtering assumptions

The DVF silver layer keeps only:

- `nature_mutation = Vente`
- `type_local` in `Maison`, `Appartement`
- strictly positive `valeur_fonciere`
- strictly positive `surface_reelle_bati`
- non-empty department code

It also removes outliers with these thresholds:

- `price_m2` between `300` and `50_000`
- `surface_reelle_bati` between `9` and `1_000`

These filters are implemented in `data/scripts/dvf/build_silver.py`.

### DVF source-family normalization

The project supports two different DVF source families:

- `2014-2020`: legacy DGFiP raw text / CSV / ZIP exports
- `2021+`: normalized Geo-DVF `full.csv.gz`

The bronze step:

- auto-detects compression
- auto-detects delimiter
- normalizes the historical and modern files into one canonical intermediate schema

This is why historical years and recent years can be queried through the same frontend table even though their original formats differ.

### FiLoSoFi harmonization assumptions

The FiLoSoFi gold layer does not invent missing indicators.

If a source does not publish a variable for a given year / geography, the harmonized gold layer keeps it `null`.

Key consequences:

- `d5_income` is intentionally kept `null` in the harmonized gold layer because there is no explicit published `D5` source column in the configured catalogue.
- `2023` commune rows often contain the median but not the full decile set.
- department series can be either:
  - official Insee department rows
  - derived aggregations from commune rows when no official department file is available

### Derived vs official department FiLoSoFi

The project keeps two department-level products:

- `department_official`: official Insee department rows when available
- `department_derived`: derived rows aggregated from communes when official rows are unavailable

They are not equivalent.

Derived department rows are explicitly flagged as:

- `indicator_source = derived_from_communes`
- `is_official = false`
- `comparable_with_previous_years = false`

### National FiLoSoFi series on the homepage

The homepage national income chart does not rely only on the local commune-derived summary.

When official national FiLoSoFi series are available, the homepage uses:

- `public/data/filosofi_national_series.json`

This was chosen to avoid presenting a commune-aggregation surrogate as if it were an official national series.

## Frontend data flows

The frontend uses two distinct mechanisms.

### 1. Static local files bundled with the site

These files are read directly from `public/data/`:

- `public/data/dvf_summary.json`
- `public/data/dvf_preview.json`
- `public/data/dvf_previews/year=YYYY/dvf_preview.json`
- `public/data/filosofi_summary.json`
- `public/data/filosofi_summaries.json`
- `public/data/filosofi_national_series.json`
- `public/data/filosofi_preview.json`
- `public/data/departements.geojson`

These files are generated by the pipelines, committed to Git, and shipped with the static frontend build.

Changing them requires a new frontend deployment.

### 2. Remote FiLoSoFi artifacts published on R2

The frontend also queries:

- `gold/filosofi/metadata.json`
- `gold/filosofi/indicator_availability.json`
- `gold/filosofi/commune_all_years.parquet`
- `gold/filosofi/department_official/department_all_years.parquet`
- `gold/filosofi/department_derived/department_all_years.parquet`

These files are not bundled in the site.

If their public URLs stay stable, updating them does not require rebuilding the frontend.

## Data Explorer behavior

The Data Explorer page mixes two modes.

### DVF mode

DVF uses a local preview workflow:

- the page loads `public/data/dvf_preview.json` or `public/data/dvf_previews/year=YYYY/dvf_preview.json`
- if a yearly public preview is missing, it can fall back to reading the remote yearly silver Parquet directly in-browser

Important nuance:

- the public preview `columns_count` describes the full source preview file
- the displayed table can expose fewer columns for some historical years if the selected year’s available parquet schema is narrower

### FiLoSoFi mode

FiLoSoFi uses a query workflow:

- metadata and availability drive the available years and indicators
- queries are executed client-side through DuckDB-Wasm on remote Parquet files

Important nuance:

- the FiLoSoFi table shown to the user is not the static `public/data/filosofi_preview.json`
- it is a live query result over the consolidated Parquet assets

## Environment variables

### Frontend

Set in `.env.local`:

```bash
VITE_DATA_ASSET_BASE_URL=https://example.r2.dev/<public-prefix>
```

This value is the public base URL used by the browser to reach FiLoSoFi artifacts.

The code trims a trailing slash automatically and appends paths such as:

```text
gold/filosofi/metadata.json
gold/filosofi/indicator_availability.json
gold/filosofi/commune_all_years.parquet
```

Do not put secrets in any `VITE_` variable.

### Server-side / workflows

Used by upload scripts and GitHub Actions:

- `R2_ACCESS_KEY_ID`
- `R2_SECRET_ACCESS_KEY`
- `R2_ENDPOINT_URL`
- `R2_BUCKET_NAME`

These must never be exposed to the frontend.

## Frontend deployment

The frontend is intended to be published on GitHub Pages at:

- `https://loic-marigny.github.io/real-estate-observatory/`

The deployment workflow is:

- `.github/workflows/deploy-pages.yml`

By default it deploys automatically on every push to `main`, and it computes the Vite base path from the repository name unless `VITE_BASE_PATH` is explicitly set in GitHub repository variables.

GitHub Pages must be enabled for the repository:

- either manually in `Settings > Pages` with `Source = GitHub Actions`
- or automatically by defining a repository secret named `PAGES_DEPLOY_TOKEN` with a token allowed to manage Pages settings for the repository
